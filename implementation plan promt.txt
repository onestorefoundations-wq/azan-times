### Project Overview

Build a comprehensive, multi-tenant **Islamic Digital Signage System** for mosques. The core is an Android TV Flutter application that runs **100% offline**, with optional cloud sync for remote management. The system must handle all edge cases of prayer time logic, including countdown timers, iqamah intervals, and special Friday prayers.

1. **Flutter TV App** — Android TV display, portrait and landscape support

### Component 1: Flutter TV App

#### 1.1 Prayer Time Calculation (Offline)

- Use the `adhan` Dart package. **Never call any external prayer time API.**
- Inputs stored in SQLite: `latitude`, `longitude`, `calculation_method`, `asr_juristic_method`, `timezone_id` (e.g., `Asia/Kolkata`).
- Calculate all 6 times (Fajr, Sunrise, Dhuhr, Asr, Maghrib, Isha) at **midnight local time** for the upcoming day using a `Timer` that reschedules itself.
- Apply `adhan_offset_mins` (positive or negative integer) per prayer after calculation.
- **Hijri Date:** Use the `hijri` Dart package to display the Islamic date alongside the Gregorian date. Recalculate at midnight.
- **Timezone handling:** Use the `timezone` Dart package. Store timezone as an IANA string. Do not rely on the device's system clock timezone alone.

#### 1.2 Prayer State Machine

The app must always know which of these states it is in. Build an explicit `PrayerState` enum and a `PrayerStateManager` service:

```
IDLE_BEFORE_FAJR       // Before first prayer of the day
PRE_ADHAN              // X minutes before any Adhan (slideshow pauses)
ADHAN_TIME             // Exact Adhan time (audio plays, full-screen alert)
IQAMAH_COUNTDOWN       // Counting down to Iqamah (adhan_offset + iqamah_wait)
PRAYER_IN_PROGRESS     // From Iqamah until Y minutes after (slideshow blocked)
POST_PRAYER            // Y minutes after Iqamah, normal display resumes
BETWEEN_PRAYERS        // Normal idle display with slideshow
SUNRISE_WINDOW         // Sunrise displayed, no prayer state changes here
```

The state manager uses a `StreamController` that the UI listens to. All visual changes (slideshow pause, countdown display, full-screen overlay) are driven by this stream.

#### 1.3 Iqamah Countdown Timer

When state enters `ADHAN_TIME`:
- Start a visible countdown on screen: `Iqamah in: MM:SS`
- Duration = `iqamah_waiting_mins` from SQLite for that prayer
- When countdown reaches 00:00, state transitions to `PRAYER_IN_PROGRESS`
- After `prayer_awareness.pause_after_iqamah_mins`, transition to `POST_PRAYER` then `BETWEEN_PRAYERS`

#### 1.4 Friday (Jumuah) Special Handling

Add a `jumuah_settings` table in SQLite:

```sql
CREATE TABLE jumuah_settings (
  id INTEGER PRIMARY KEY,
  khutbah_time TEXT,       -- e.g., "13:00" local time
  iqamah_time TEXT,        -- e.g., "13:30"
  display_label TEXT       -- e.g., "Jumu'ah"
);
```

On Fridays, replace the Dhuhr row in the prayer table UI with the Jumuah row. The state machine applies the same `PRE_ADHAN → ADHAN_TIME → IQAMAH_COUNTDOWN` logic using these times.

#### 1.5 Taraweeh Display

When `features_format.show_taraweeh = true`, add a **sixth row** in the prayer table below Isha labeled **"Taraweeh"**. This row shows a static time stored in `taraweeh_time` in SQLite. No calculation is needed. It does not trigger the prayer state machine.

#### 1.6 Audio Alerts

- Store a short audio file (e.g., a 5-second notification chime, **not** a full Adhan recitation unless specifically configured) bundled inside the Flutter assets folder at `assets/audio/alert.mp3`.
- Use the `audioplayers` Dart package.
- Play on `ADHAN_TIME` state entry if `audio_alerts_enabled = true`.
- Provide a separate boolean `use_full_adhan_recitation` for mosques that want the full Adhan audio file (user must supply their own MP3 and it is stored locally).

#### 1.7 UI Layout — Two Modes

**Landscape (default):**
- Left 60%: Prayer time table (all 6 prayers in rows, highlight the upcoming prayer)
- Right 40%: Current time (large), Hijri + Gregorian date, upcoming prayer name + countdown, Iqamah countdown when active
- Bottom bar: Scrolling announcement ticker (marquee)
- Slideshow: overlays the right 40% panel when active in `SPLIT_SCREEN` mode, or full-screen in `FULL_SCREEN` mode

**Portrait:**
- Top 35%: Current time (large), Hijri + Gregorian date
- Middle 50%: Prayer table
- Bottom 15%: Scrolling announcement ticker
- Slideshow: overlays full screen in both modes (split screen is not applicable in portrait)

Both layouts must use `MediaQuery` + `OrientationBuilder`. Font sizes must be defined using `MediaQuery.of(context).size` so they scale correctly to different TV resolutions (720p, 1080p, 4K).

**Typography rules:**
- All Arabic text (masjid name, prayer names if configured in Arabic) must use a bundled Arabic font (`assets/fonts/NotoNaskhArabic-Regular.ttf`) with correct RTL text direction.
- Provide a `use_arabic_labels` boolean in SQLite `features_format`. When true, prayer names render in Arabic script.
- Time display uses `intl` Dart package, respecting the `use_24_hour_format` setting.

#### 1.8 Slideshow Engine

Behavior rules (all configurable per `slideshow_settings` in SQLite):

| Setting | Type | Description |
|---|---|---|
| `enabled` | bool | Master on/off |
| `interval_minutes` | int | How many minutes between slideshow runs |
| `duration_per_image_seconds` | int | How long each image is shown |
| `display_mode` | enum | `split_screen` or `full_screen` |
| `transition_type` | enum | `fade`, `slide`, `none` |
| `pause_before_adhan_mins` | int | Block slideshow this many minutes before Adhan |
| `pause_after_iqamah_mins` | int | Block slideshow this many minutes after Iqamah |

- Images are stored in the device's local storage at a defined path (e.g., `/data/user/0/<package>/files/slides/`).
- The slideshow engine must check `PrayerStateManager` before showing each image. If state is `PRE_ADHAN`, `ADHAN_TIME`, `IQAMAH_COUNTDOWN`, or `PRAYER_IN_PROGRESS`, **silently skip** and do not restart the timer.
- Resume the regular schedule when state returns to `BETWEEN_PRAYERS`.

#### 1.9 Embedded Local Admin Panel (HTTP Server)

- Use the `shelf` and `shelf_router` Dart packages.
- Bind to `0.0.0.0` on port **8080** when the app starts.
- **Security:** Generate a 4-digit PIN stored in SQLite at first launch. Display it on the TV screen in settings. The HTML admin page must prompt for this PIN before showing any controls. Use session tokens (stored in memory) with a 1-hour expiry.
- The admin panel serves a single-page HTML/JS app (embedded as a Dart string or asset) that provides:
  - View and edit all SQLite settings (profile, adjustments, features, slideshow, announcements)
  - Upload images directly from phone → TV (multipart form POST, saved to slides directory)
  - Delete individual slides
  - Trigger a "force sync from Supabase" action
  - View the current `PrayerState` and next prayer countdown
- The server must only accept connections from RFC-1918 private IP ranges. Reject all other IPs.

### Component 3: Supabase Schema (Multi-Tenant)

All tables include `tenant_id UUID REFERENCES tenants(id)` with Row Level Security enabled. Admins only see their own rows.

```sql
CREATE TABLE tenants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE mosque_configs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  config_version INTEGER NOT NULL DEFAULT 1,
  config_json JSONB NOT NULL,    -- Full config matching the SQLite schema above as JSON
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  updated_by TEXT                -- Super admin email
);

CREATE TABLE device_registry (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id),
  device_id TEXT NOT NULL UNIQUE, -- Matches sync_meta.device_id on TV
  last_seen TIMESTAMPTZ,
  app_version TEXT,
  online_status BOOLEAN DEFAULT FALSE
);
```

**Sync logic (Flutter background worker):**
- On internet availability (use `connectivity_plus` package), fetch `mosque_configs` where `tenant_id` matches and `config_version > local supabase_config_version`.
- If a newer config exists, deserialize the `config_json` and write each table's fields into SQLite, updating `last_modified` to the current timestamp.
- **Conflict rule:** Cloud config wins for all settings EXCEPT `local_admin_pin` and `device_id`, which are never overwritten by cloud sync.
- After writing, update `sync_meta.supabase_config_version` and `last_successful_sync`.
- Update `device_registry.last_seen` and `online_status` on every sync attempt (pass or fail).

### Component 4: React Super Admin Dashboard (Vite)

**Pages:**

1. **Dashboard** — Grid of mosque cards showing name, location, online/offline status (from `device_registry.last_seen`), last sync time
2. **Mosque Profile Editor** — Edit all fields mapping to `mosque_configs.config_json`. Includes a live preview of how the prayer table will look.
3. **Announcements Manager** — Add/edit/reorder/delete announcements with Arabic text support
4. **Asset Manager** — Upload images via `uploads.php`, display thumbnails, drag to reorder, delete. Show the remote URL returned from PHP server.
5. **Push Config** — Button to increment `config_version` and save, triggering TV devices to pull on next sync check.

**PHP Asset Server (`uploads.php`) spec:**
- Accept `multipart/form-data` POST with field `file`
- Validate: image types only (`jpg`, `png`, `webp`, `gif`), max 5MB
- Save to a configurable `$upload_dir` on the server
- Return JSON: `{ "success": true, "url": "https://your-server.com/uploads/filename.jpg" }`
- Include basic API key authentication via `Authorization: Bearer <key>` header
